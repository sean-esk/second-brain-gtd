**ALL Second Brain operations MUST work in the vault folder.**

**To find the vault folder:**

```json

  "vaultFolder": "General",
  "setupDate": "2025-10-25",
  "setupComplete": true,
  "userName": "Sean"
}
```

**This means:**
- If vaultFolder = "General", use `General/00-Inbox/Daily/` (not `ObsidianVault/00-Inbox/Daily/`)
- If vaultFolder = "MyVault", use `MyVault/00-Inbox/Daily/`
- etc.

**NEVER hardcode "ObsidianVault/" paths in your responses.**

**How to use:**
1. At start of ANY Second Brain command, read `.claude/vault-config.json`
2. Extract `vaultFolder` value
3. Use that in all paths: `{{vaultFolder}}/00-Inbox/Daily/`

**If vault-config.json doesn't exist:**
- User hasn't run `/setup` yet
- Prompt them: "Please run `/setup` first to configure your Second Brain"
- Do not proceed

**This is configured during `/setup` command (Step 0) and ensures all files are saved in the correct location.**

## Key Principles

### ADHD-Friendly Editing

**The system is designed for users who may have ADHD. When editing documents:**

1. **Read the entire document first** - Understand existing structure and content
2. **Make targeted edits** - Update specific sections, don't append new ones
3. **Never just add to the bottom** - Unless explicitly asked to append
4. **Keep it concise** - Remove redundancy, don't create multiple versions of the same section
5. **One plan, not many** - When updating plans, replace the old plan, don't add "revised plan" sections

**Bad example:** Adding "REVISED PLAN" section below "TODAY'S PLAN" section
**Good example:** Replacing "TODAY'S PLAN" content with updated tasks checked off

This is critical for managing ADHD. Bloated documents with multiple versions of the same information are counterproductive and overwhelming.

## Custom Commands

The system provides AI-powered slash commands for workflow automation.

### Core Workflow Commands

#### `/setup`
**Purpose:** Interactive onboarding to configure the system
**When:** First-time setup or system reset
**What it does:**
- Asks about user's goals (6-month, 3-month, 1-month)
- Asks about areas of responsibility
- Creates `Assisting-User-Context.md` permanent note
- Sets up folder structure
- Provides onboarding guidance

#### `/capture`
**Purpose:** Get it out of your head and into the system (GTD Capture stage)
**When:** Anytime you have a thought, task, or idea
**What it does:**
- Appends ALL captures to `ObsidianVault/00-Inbox/Daily/YYYY-MM-DD.md`
- No categorization, no routing - just capture with timestamp
- Works for single items OR full brain dumps
- **GTD Principle:** "Capture first, clarify later"

#### `/process-inbox`
**Purpose:** Clarify, organize, and review (GTD Clarify + Organize + Reflect stages)
**When:** 3x per week minimum
**What it does:**
- Asks clarifying questions FIRST for vague items
- Applies GTD clarifying questions to each capture
- Routes to appropriate locations:
  - Actionable â†’ Projects/Tasks
  - Knowledge â†’ Fleeting notes (for later development)
  - Reference â†’ Reference notes
- Reviews all active projects (mini weekly review)
- Archives completed projects
- Updates _Context.md

#### `/daily-plan`
**Purpose:** Choose what to work on today (GTD Engage stage)
**When:** Every morning
**What it does:**
- Checks inbox first (prompts to run `/process-inbox` if 5+ items)
- Handles "no projects" scenario (guides user to create projects)
- Reads all active projects and extracts next actions
- Asks about energy, context, time available (GTD four criteria)
- Generates prioritized task list (must-do, should-do, quick-wins)
- Creates/enhances plan in `ObsidianVault/Daily Plans/YYYY-MM-DD.md`
- If draft plan exists from closeout, enhances it with energy/context
- Updates `_Context.md`

#### `/daily-closeout`
**Purpose:** Daily reflection and tomorrow prep (GTD Reflect stage)
**When:** Every evening
**What it does:**
- Reads today's plan from `Daily Plans/`
- Asks what was accomplished
- Marks completed/partial/deferred items
- Asks about tomorrow's priorities (optional)
- Creates tomorrow's DRAFT plan (priorities without energy/context)
- Clarifies this is a draft - will be refined by `/daily-plan` in morning
- Updates `_Context.md`

## Working with This Vault

### When Assisting Users

1. **Respect ADHD-friendly editing** - Read entire documents before editing, make targeted updates
2. **Reference user context** - Check `Assisting-User-Context.md` to understand priorities
3. **Use PARA structure** - Place notes in appropriate actionability category
4. **Follow Zettelkasten principles** - Atomic notes, own words, 3+ links
5. **Maintain frontmatter** - Include date, tags, status, type on all notes
6. **Be encouraging** - Positive, helpful tone without judgment
7. **Explain reasoning** - Help users understand why suggestions are made

### Note Creation Guidelines

**Frontmatter (required):**
```yaml
---
title: "Note Title"
created: "YYYY-MM-DD"
tags:
  - tag1
  - tag2
status: active/draft/completed
type: project/area/resource/permanent-note/etc
---
```

**File Naming:**
- Projects: Descriptive names (`Website-Redesign.md`)
- Daily Plans: `YYYY-MM-DD.md`
- Permanent Notes: Descriptive concept (`Context-Switching-Reduces-Focus.md`)
- Meeting Notes: `YYYY-MM-DD - Topic.md`
- MOCs: `MOC - Topic.md`

**Linking:**
- Use `[[double brackets]]` for internal links
- Minimum 3+ links for permanent notes (no orphans)
- Explain relationships in link context

**Tags:**
- Use hierarchical tags: `#status/active`, `#context/computer`
- Type tag required: `#project`, `#permanent-note`, etc
- Business/area tags: User-defined during setup
- Status tags: Track workflow state
